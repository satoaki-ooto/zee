# Roadmap: zee — 矩形（ブロック）選択・編集モード

## 向かう先
zee に矩形（ブロック）選択モードを入れ、Emacs の rectangle 相当を扱えるようにする。
すなわち矩形での **選択・コピー・カット・ペースト・矩形挿入** のすべてを、通常の編集と同じ操作感（1操作＝1 undo）で行える状態を最終地点とする。

## 現在の区間: 矩形のカット/削除まで（編集モデルの土台を変える区間）

向かう先のうち、まず「読み取り（選択・ハイライト・コピー）」に加えて「破壊的編集（カット/削除）」までを一気に取りに行く。
カット/削除は単一連続領域前提の編集・undo モデルに正面からぶつかるため、この区間の重心は **編集モデルそのものの変更** にある。ペースト系は土台が変わってから次の区間で乗せる。

### この区間でやること
- 矩形選択モードに入り、(行, 列) の2次元範囲を選べる
- 矩形選択範囲が画面上でハイライトされる
- 矩形選択範囲をコピーできる（行ごとに切り出して連結）
- 矩形選択範囲をカット／削除できる。かつ **1回の undo で矩形操作全体が戻る**
- 上記カット/削除を成立させるため、単一連続領域を前提とした編集・undo モデル（`OpaqueDiff` / `EditTree` の revision）を、**不連続な複数領域の編集を1操作として扱える**ように変更する ← この区間の中心
- Shift modifier を持たない zi のキー入力制約の下で、矩形モードのキーバインド方式を確定する

### この区間でやらないこと
- 矩形ペースト・矩形挿入（行ごとに挿入位置をずらす挿入） — 次の区間。編集モデルを変えた土台の上に乗せる
- clipboard が「矩形で切った」というメタ情報を保持すること — 矩形ペーストとセットなので次の区間に回す
- 複数カーソルと矩形選択の併用 — 現状は単一カーソル運用であり、そもそも今は作らない
- tab / CJK 幅をまたぐ列計算の網羅的なエッジケース対応 — 基本の幅整合（grapheme width 準拠の列マッピング）まではこの区間に含む。変則ケースの作り込みは後回し
- 各能力の「どう動作すべきか」「どう作るか」の詳細設計 — behavior-definition → requirement-definition → openspec の領域。このロードマップは方向と範囲までで止める

### 状態
- **完了** ─ 2026-06-13 実装完了。compound diff 機構（`CompoundDiff`：`OpaqueDiff` の順序付き列を1リビジョンに束ねる、D1）と矩形座標型（`RectangleSelection`）を導入。キーバインド方式は `Ctrl-x r` プレフィックス（D6）で確定。通常編集の後方互換（C-3）を構造的に担保（通常編集＝1要素列の退化ケース）。

## これまでの区間（要約）
- **矩形のカット/削除まで**（完了）: 編集・undo モデルを不連続複数領域対応に拡張。矩形選択→ハイライト→コピー→カット/削除まで実装。compound diff 機構（D1）と `Ctrl-x r` プレフィックスキーバインド（D6）を確定。

## 次の区間の候補
- 矩形ペースト / 矩形挿入、および clipboard の「矩形」メタ情報の保持
- tab / CJK 幅をまたぐ列計算のエッジケース対応
- （必要になれば）複数カーソルと矩形選択の統合

## 残された曖昧さ
- ~~矩形カット/削除の undo を「1操作＝1リビジョン」で実現する具体手段が未決~~ → **解決**: `CompoundDiff`（`OpaqueDiff` の順序付き列）を採用。バウンディング近似は C-2 違反で却下。
- ~~矩形モードのキーバインド具体割り当てが未確定~~ → **解決**: `Ctrl-x r` プレフィックス方式（`Ctrl-x r Space` で進入、`Ctrl-x r M-w` でコピー、`Ctrl-x r C-w` でカット、`Ctrl-x r d` で削除）。
- ~~既存 `reconcile()` の次元混在疑い~~ → compound 対応で `reconcile_compound()` を追加。単一 `reconcile()` は不変。次元混在は現状踏まず（単一カーソル運用で影響限定的）。
