# 要件定義の結果: 矩形ペースト（yank-rectangle）

> ワーキングメモリ（短寿命）。実装サイクルが一区切りしたら、観察は work-journal に引き継ぎ、このファイルは次スコープで上書き／破棄する。
> 紐づく上位: ROADMAP「矩形ペースト（貼り戻し）まで」区間 / `docs/behavior/rectangle-paste.md`。

## もとの上位要件

- **何のためにあるか**: 矩形 kill したもの（矩形コピー/カットの結果）を、矩形として下方向に貼り戻せる（yank-rectangle）。
- **何が解けたら終わりか**: `Ctrl-x r y` 系で矩形ペーストができる（3変種・短い行はスペース padding・末尾で行追加・矩形kill無しはno-op・1操作=1undo）。前提として clipboard が「矩形 kill」を通常 kill と区別し、各行を復元できる。

## 実装スコープ（今回やる）

切り出し方針: 「矩形 kill ストア（土台）」を先に切り、その上に「基本ペースト」、最後に「カーソル3変種」。依存順は **S1 → S2 → S3**。

- **S1. 矩形 kill ストア（clipboard が矩形 kill を区別・各行復元できる）**
  - 矩形コピー/カットの結果を「矩形 kill」として保存し、「直近の kill が矩形 kill か」を判定でき、矩形 kill の各行を復元できる。通常 kill（フラット文字列）とは区別する。
  - 紐づく上位要件: behavior「矩形 kill が無いとき no-op」「各行を下方向に貼り戻す」の前提。通常 `Ctrl-y` の振る舞いを変えない（C-3）。

- **S2. 基本 yank-rectangle（左上カーソル変種・`Ctrl-x r y`）**
  - 矩形 kill の各行を、カーソルの visual column `c0` を左端として下方向の連続行へ1行ずつ貼り戻す。短い行は `c0` までスペース padding。カーソル以下の行が足りなければ改行を追加して新規行を作る。矩形 kill が無ければ no-op。ペースト全体が **1操作=1undo**。挿入後カーソルは左上角（元位置）。
  - 紐づく上位要件: behavior「基本のペースト」「バッファ末尾で行不足」「短い行への padding」「undo 原子性(C-1)」「矩形 kill が無いとき no-op」。

- **S3. カーソル3変種（`Ctrl-x r Y`=右下 / `Ctrl-x r M-y`=下端の次行・左端列）**
  - S2 の挿入結果に対し、最終カーソル位置を変える2変種を追加。`Y`=最終行に貼った文字列の末尾、`M-y`=下端の一つ下の行の `c0`（次行が無ければカーソルのために改行を足さず下端行末尾）。
  - 紐づく上位要件: behavior「カーソル最終位置の3変種」。

## 保留（次の実装スコープ／次区間に回す）

- **tab をスペース展開して挿入列を厳密一致させる** — 今回は基本の幅整合（grapheme 境界丸め, C-4 範囲）まで。厳密一致は既存文字の replace を要し compound へ波及するため後回し（behavior 確定事項の根拠どおり）。
- **string-rectangle（同一列への文字列一括挿入）** — ROADMAP 次区間。yank-rectangle の土台ができてから。
- **C-5「パディング側」を `_constraints.md` へ昇格** — 現状 paste 専用。string-rectangle も padding を要する段階で昇格し重複を畳む。
- **SystemClipboard 経由での矩形性の往復（外部コピーとの相互運用の作り込み）** — 下の「残された曖昧さ」と同根。今回は内部の矩形 kill ストア成立までで、外部往復の忠実度は次に回す候補。

## 対象外（今回は作らない）

- **挿入プレビュー描画** — ROADMAP「やらない」。選択ハイライトで足りる。
- **複数カーソルと矩形ペーストの併用** — ROADMAP「やらない」。単一カーソル運用。
- **通常 `Ctrl-y`（フラット yank）の振る舞い変更** — C-3。矩形ペーストは別コマンドとして独立させ、通常 paste には一切触れない。

## 停止条件メモ（観察用）

- S2 をさらに「padding 無しの素の複数行挿入」「padding」「末尾行追加」へ割ろうとしたが、**止めた**。各断片は単体で「矩形として貼り戻せる」という上位要件に紐づかず（部分ペーストは成果として観測できない）、トレーサビリティが切れるため。S2 は behavior の1 GWT に対応する最小の塊として保持。
- S1 と S3 は紐づく一行が即書けたので、ここで切り出しを止めた。
- 「insert方向 compound（複数行挿入の1リビジョン化）」は S2 内部の最大リスクだが、これは**実装手段**なので requirement では割らず、実装プラン／openspec design に送る。

## 残された曖昧さ

- **S1 の射程＝外部クリップボードとの相互作用**: 矩形コピー後に別アプリで別の内容をコピーして戻ったとき、矩形 kill ストアが古い矩形を「直近の矩形 kill」と誤判定しないか。内部ストアのみ（外部操作で矩形性を無効化しない）で今回成立とみなすか、外部コピー検知まで含めるかが未決。→ S1 着手前にもう一周、または design へ判断を送る。

## 次にどうする

- S1 → S2 → S3 の順で実装プラン／openspec へ。まず **S1（矩形 kill ストア）** を openspec の change として起こし、その design で「メタ情報の持ち方（trait拡張/内部矩形kill ring/マーカー埋め込み）」と上の残された曖昧さ（外部クリップボード相互作用）を最初に詰める。
- S2 の「insert方向 compound」は design 最大の論点。S1 完了後の change で扱う。
