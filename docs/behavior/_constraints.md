# 横断制約: 矩形（ブロック）編集モデル

複数の能力（選択 / ハイライト / コピー / カット）にまたがって常に効く前提を、ここに独立して置く。
個別ファイルに埋もれさせない。各能力ファイルはこの制約を満たす前提で書かれる。

対象区間: ROADMAP「矩形のカット/削除まで（編集モデルの土台を変える区間）」。
この区間の重心＝編集モデルそのものの変更。よってこの制約が最上位に立つ。

> このファイルは「どう動くべきか／何は絶対起きてはいけないか」までで止める。
> 「どのコード・どの設計で実現するか」（`OpaqueDiff` を複合化するか、revision に複数 diff を束ねるか等）は openspec design の領域。

---

## C-1. 1操作 = 1リビジョン（undo の原子性）

- **常に満たす前提**: 矩形カット／矩形削除は、何行にまたがろうと **1回の undo でその矩形操作全体がちょうど元に戻る**。redo も同様に1回で全体が再適用される。
- **絶対にこうならない**:
  - undo を1回押したのに、矩形の一部の行だけが戻り、他の行が残る（部分 undo）。
  - 矩形操作の途中状態（行をN行処理した時点）が undo 履歴に独立したリビジョンとして現れる。
  - 矩形カット1回に対して、undo を複数回押さないと元に戻らない。

## C-2. 不連続な複数領域を1操作として扱える

- **常に満たす前提**: 矩形編集は、行ごとに分かれた **不連続な複数の文字範囲** をまとめて1つの編集操作とみなす。編集モデル（diff / revision / reconcile）は、この「1操作 = 複数の不連続領域の変更」を表現・適用・巻き戻しできる。
- **絶対にこうならない**:
  - 不連続な範囲を、それらを内包する単一バウンディング領域で近似した結果、**範囲外（矩形に含まれない文字）が巻き込まれて変更・消失する**。
  - 既存の単一連続編集（通常の入力・削除）の振る舞いが、このモデル変更によって変わる。通常編集は今までどおり動く。

## C-3. 既存の単一連続編集との後方互換

- **常に満たす前提**: 通常の1文字入力・連続範囲の選択/削除/コピーは、この区間の変更後も **観測可能な振る舞いが一切変わらない**。
- **絶対にこうならない**: 矩形対応のために、通常編集の undo 粒度・カーソル追従・選択正規化が変わる。

## C-4. 列は visual column（grapheme 幅準拠）で測る

- **常に満たす前提**: 矩形の「列」は char index ではなく **表示上の視覚列**。CJK は幅2、tab は `tab_width` 分として扱い、`graphemes::width()` / sticky column (`visual_horizontal_offset`) と同じ尺度で列を測る。これにより、見えている矩形と実際に切り取られる範囲が一致する。
- **絶対にこうならない**: 画面上でハイライトされている矩形と、コピー/カットされる実際の文字範囲がずれる。
- **この区間の範囲**: 基本の幅整合（grapheme width 準拠の列マッピング）まで。tab / CJK を列の途中でまたぐ変則ケースの作り込みは次区間（ROADMAP「やらないこと」）。

## C-5. 短い行（ragged line）の扱いは差し替え可能な分岐点

- **この区間の決定**: 矩形の列範囲より短い行は **Emacs 準拠＝その行分は「空」として扱う**（空白パディングはしない）。詳細は各能力ファイル（copy / cut）に記す。
- **常に満たす前提（申し送り）**: ragged line のポリシー（空扱い ↔ 空白パディング）は、**後から差し替えられる単一の分岐点として隔離**されていること。コピー・カット・ハイライトの各所にこの判断が散らばってはならない。
  - これは「空白パディングに変えたくなったときに1箇所で切り替えられる」ことを保証するための制約。具体的な隔離手段（関数・型・trait のどれにするか）は openspec design へ。

---

## 残された曖昧さ（横断レベル）

- C-1 の「1操作 = 1リビジョン」を満たす具体機構（複合 diff / revision に複数 diff を束ねる / 単一バウンディング diff で近似）は **未決**。ただしバウンディング近似は C-2 の「絶対にこうならない」に抵触しうるため、採るなら範囲外巻き込みが起きないことの担保が要る。→ openspec design で詰める。
- `reconcile()` の `char_index..max(old_char_length, new_char_length)`（rectangle-mode.md の指摘）が、複数領域 reconcile で次元混在の脆さを踏むか **未確認**。この区間で触る必要が出るかは実装着手時に判定。

## 次にどうする

- 各能力ファイル（selection / highlight / copy / cut）はこの制約を前提に書く。
- C-1 の機構未決は、この区間の中心の不確実性。requirement-definition で「最初に潰すスコープ」として切り出す候補。
