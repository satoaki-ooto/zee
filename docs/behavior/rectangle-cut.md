## 振る舞い定義: 矩形カット / 矩形削除

### 何のための対象か

ROADMAP「矩形のカット/削除まで」区間の**重心**。
矩形選択範囲を各行から取り除き、左右に残った部分を繋ぐ。これを **1回の undo で全体が戻る** 1操作として成立させる。横断制約 C-1（undo 原子性）・C-2（不連続領域を1操作）・C-4（visual column）・C-5（短い行は Emacs 準拠＝空扱い）を正面から満たす対象。

カット = 取り除いた内容を clipboard に入れてから削除。削除 = clipboard に入れず削除。**編集としての振る舞いは両者同一**で、差は clipboard への書き込み有無のみ。

### 期待する振る舞い

濃度は Given/When/Then（破壊的編集かつ undo 原子性が中心。最も硬く固定する）。

- **基本のカット（GWT）**
  - Given: 矩形が行範囲 `r0..=r1`、visual column `[left, right)` で選択され、各行に該当文字がある
  - When: 矩形カット操作をする
  - Then:
    1. （カットのみ）各行の `[left, right)` 部分を copy と**同一規則**で切り出し、`\n` 連結して clipboard に set する。
    2. 各行 r で、その行の `[left, right)` に当たる文字を削除し、`[0, left)` と `[right, 行末]` を連結する（矩形の穴が閉じる）。
    3. この複数行の削除が **1リビジョン**として履歴に積まれる。
    4. 矩形選択は解除され、カーソルは矩形の**左上角**（`r0`, visual column `left`）に置かれる。【確定: Emacs `kill-rectangle` 準拠】

- **削除（GWT）**: 上記の 1.（clipboard 書き込み）を行わない。2.〜4. は同一。clipboard は変化しない。

- **undo 原子性（GWT、C-1）**
  - Given: 矩形カット/削除を1回行った直後
  - When: undo を1回する
  - Then: 全行の矩形削除が同時に巻き戻り、バッファは矩形操作の直前と完全一致。続けて redo を1回すると矩形削除が全行同時に再適用される。

- **短い行（C-5: Emacs 準拠＝空扱い、散文）**
  - 行末 < `left` の行 → その行は**何も削除しない**（空白で詰めて揃えたりしない）。
  - 行末が `[left, right)` の途中までの行 → `[left, 行末]` までの存在する文字だけ削除する。

### 許さない振る舞い・守る制約

- **絶対にこうならない**（C-1 / C-2 の具体化）:
  - undo 1回で矩形の一部の行だけが戻る／全体を戻すのに undo を複数回要する（C-1）。
  - 矩形に含まれない文字（`[0,left)` や `[right,行末]`、選択行範囲外の行）が削除・改変される。バウンディング近似で範囲外を巻き込む（C-2）。
  - カットの clipboard 書き込みは済んだのに削除が一部失敗し、clipboard とバッファが食い違う中途半端状態が残る（操作は全行成功か、全体取り消しか）。
  - 削除される列範囲が、ハイライト/コピーで見えていた範囲とずれる（C-4）。
  - 短い行を空白でパディングしてから削除し、選択していない位置の空白構造を壊す（C-5 の空扱いに反する）。
- **常に満たす前提**:
  - 通常の連続削除（既存 `delete_selection`）の undo 粒度・カーソル挙動は変わらない（C-3）。
  - 切り出し/列計算の規則は copy・highlight と共有（3者で分岐しない）。C-5 の空扱い↔padding 切替は共有の分岐点経由。

### 濃度

最も厳密(GWT)。理由: この区間の重心であり、破壊的かつ undo 原子性という「越えたら戻れない」制約を持つため、正常系も禁止系も硬く固定する。

### 確定事項（ユーザ確認済み）

- **カット後カーソル**: 矩形の左上角（`r0`, visual column `left`）。Emacs `kill-rectangle` 準拠。
- **幅0矩形のカット/削除**: **no-op**（バッファ・clipboard・undo 履歴のいずれも変化しない）。padding しない（_constraints C-5）。

### 残された曖昧さ（振る舞いレベル）

- なし。

### design へ送る実装論点（振る舞いではない）

- **1リビジョン化の具体機構**（C-1 と同根、この区間最大の不確実性）: 複合 diff / revision に複数 diff を束ねる / 単一バウンディング diff で近似。バウンディング近似は「範囲外を巻き込まない」（C-2 禁止系）を別途担保する必要があり、衝突しうる。→ openspec design の最初の論点。
- カットと削除の **UI 操作（キー）を別々に用意するか**。編集としての振る舞いは同一と確定済み、キーは rectangle-selection のキーバインド方式と合わせて design で確定。
- `reconcile()` の既存の脆さ（rectangle-mode.md）を複数領域削除で踏むか未確認。踏むならこの区間で触る。

### 次にどうする

- selection / highlight / copy と合わせて4枚を requirement-definition へ（振る舞いゲートは閉じた）。
- 「1リビジョン化の機構」は実装手段だが**この区間最大の不確実性**。requirement-definition で最初に潰すスコープとして切り出すことを推奨。
