## 振る舞い定義: 矩形ペースト（yank-rectangle）

### 何のための対象か

ROADMAP「矩形ペースト（貼り戻し）まで」区間の**重心**。
矩形 kill したもの（矩形コピー / 矩形カットの結果）を、矩形として下方向に貼り戻す。各行を、カーソルの visual column を左端として、下に連続する行へ1行ずつ挿入し、これを **1回の undo で全体が戻る** 1操作として成立させる。前区間の横断制約 C-1（undo 原子性）・C-2（不連続領域を1操作）・C-3（後方互換）・C-4（visual column）・C-5（短い行ポリシーは差し替え可能な分岐点）の上に乗る対象であり、C-5 については **paste 側で「空白パディング」の分岐を採る**（copy/cut の「空扱い」とは逆側）。

この区間では yank-rectangle のみを取りに行く。矩形挿入（string-rectangle）は次区間。

### 期待する振る舞い

濃度は Given/When/Then（破壊的編集＝挿入かつ undo 原子性が中心。前区間の C-1/C-2 の土台に乗るため硬く固定する）。

- **基本のペースト（GWT）**
  - Given: 直近の kill が**矩形 kill**であり（矩形コピー/カット由来）、カーソルが visual column `c0` にある
  - When: 矩形ペースト操作をする
  - Then:
    1. 矩形 kill の各行 `L0, L1, … Ln` を、カーソル行から下方向の連続する行 `r0, r0+1, …, r0+n` の、それぞれ visual column `c0` を左端として挿入する。
    2. 各行で、`c0` に当たる既存文字の手前にその行のテキストを差し込む。`c0` 以降に既存テキストがある場合、それは右へずれる（消えない）。
    3. この複数行への挿入が **1リビジョン**として履歴に積まれる。
    4. 挿入後のカーソル位置は変種により決まる（後述「カーソル最終位置の3変種」）。

- **バッファ末尾で行が足りない（GWT）**
  - Given: カーソル以下に残る行数が、矩形 kill の行数より少ない
  - When: 矩形ペーストをする
  - Then: 足りない行は**改行を追加して新しい行を作り**、矩形 kill の全行を貼る（Emacs `yank-rectangle` 準拠）。

- **短い行への padding（C-5 の paste 側分岐＝空白パディング、散文）**
  - 挿入先の行末 visual 幅が `c0` 未満（行が `c0` に届かない）→ その行を **`c0` まで半角スペースで詰めてから**、矩形 kill のその行ぶんを挿入する。
  - 挿入列 `c0` が既存グラフェム（CJK 幅2 / tab）の途中に当たる → **基本の幅整合（grapheme 境界へ丸める）まで**で挿入する（C-4 のこの区間範囲）。tab をスペースへ分割展開して列を厳密一致させる作り込みはこの区間では**やらない**（実装裁量／後回し）。

- **カーソル最終位置の3変種（GWT、キーで選ぶ）**
  - 基本＝**左上角**（`Ctrl-x r y`）: カーソルはペースト前の位置（`r0`, visual column `c0`）＝貼った矩形の左上に留まる。Emacs `yank-rectangle` 準拠。
  - **右下**（`Ctrl-x r Y`）: 最終行 `r0+n` に貼った文字列の**末尾**（ragged なまま、追加 padding なし）。
  - **下端の次行・左端列**（`Ctrl-x r M-y`）: 貼った矩形の下端 `r0+n` の**一つ下の行**の visual column `c0`。下端が既にバッファ最終行で次行が無い場合は、カーソル配置のためだけに改行を足さず、下端行の末尾に置く（後述「許さない振る舞い」）。

- **undo 原子性（GWT、C-1）**
  - Given: 矩形ペーストを1回行った直後
  - When: undo を1回する
  - Then: 全行への挿入（および padding・末尾の行追加を含む）が同時に巻き戻り、バッファはペースト直前と完全一致。続けて redo を1回すると全体が同時に再適用される。

- **矩形 kill が無いとき（散文）**
  - 直近の kill が矩形 kill でない（通常コピー / 外部コピー / 何も kill していない）→ 矩形ペーストは **no-op**（バッファ・clipboard・undo 履歴のいずれも変化しない）。矩形ペーストは矩形 kill 専用（Emacs の `killed-rectangle` 準拠）。

### 許さない振る舞い・守る制約

- **絶対にこうならない**:
  - undo 1回で一部の行への挿入だけが戻る／全体を戻すのに undo を複数回要する（C-1）。
  - ペースト途中状態（N行貼った時点）が独立したリビジョンとして履歴に現れる（C-1）。
  - 挿入位置 `c0` 以外の既存文字が削除・改変される。`c0` 以降の既存テキストは右へずれるだけで、**消えたり書き換わったりしてはならない**（C-2 / 挿入の純粋性）。
  - 見えている（カーソルがいる）挿入列と、実際に挿入される列がずれる（C-4）。
  - **カーソルを目的位置へ置くためだけに、改行や空白をバッファに足す**（M-y 変種で次行が無い場合に改行を勝手に作る等）。バッファ変更は「矩形 kill を貼る」ためにのみ起き、カーソル移動の都合では起きない。
  - 矩形 kill が無いのに何かを挿入する（no-op 違反）。フラット文字列を勝手に矩形分割して貼る、通常 paste にフォールバックして貼る、等もしない。
  - 矩形ペーストの実装が、通常 paste（`Ctrl-y` のフラット挿入）の振る舞いや、clipboard 文字列そのものを壊す（C-3）。
- **常に満たす前提**:
  - 通常の `Ctrl-y`（フラット yank）の挿入位置・undo 粒度・カーソル挙動は、この区間の変更後も一切変わらない（C-3）。矩形ペーストは通常 paste とは**別コマンド**として独立する。
  - 列計算（visual column → char index）の尺度は highlight / copy / cut と共有（C-4、3者で分岐しない）。
  - C-5 の「空扱い ↔ 空白パディング」の切替は、隔離された単一の分岐点を経由する（各所に散らさない）。paste はその分岐点の「パディング」側を使う。

### 濃度

厳密(GWT)。理由: 破壊的編集（複数行への不連続挿入）であり、前区間で確定した C-1（undo 原子性）・C-2（不連続領域を1操作）の上に正面から乗るため。正常系も禁止系も硬く固定する。

### 確定事項（ユーザ確認済み）

- **射程**: yank-rectangle のみ。string-rectangle（同一列への文字列一括挿入）は次区間。
- **末尾で行不足**: 改行を追加して新しい行を作り、全行を貼る（Emacs 準拠）。
- **カーソル最終位置**: 3変種をキーで選ぶ。`Ctrl-x r y`=左上角（基本）、`Ctrl-x r Y`=右下（最終行に貼った文字列の末尾）、`Ctrl-x r M-y`=下端の次行・左端列。
- **「右端」の定義**: ragged なクリップボード前提で、右下変種の右端は「最終行に貼った文字列の末尾」（追加 padding はしない）。
- **キー競合**: `Ctrl-x r` 配下の既存は `Space / M-w / C-w / d` のみ。`y / Y / M-y` は空きで競合なし。zi の `Key` は Shift を独立修飾キーに持たず大文字は `Char('Y')` として表現可能（実コード確認済み）。
- **矩形 kill が無いとき**: no-op（矩形ペーストは矩形 kill 専用）。
- **padding / tab**: 短い行はスペースで `c0` まで詰めて挿入する。挿入列が tab/CJK の途中に当たる場合は基本の幅整合（grapheme 境界へ丸める、C-4 範囲）まで。tab をスペース展開して厳密一致させる作り込みはこの区間では行わない（後回し／実装裁量）。
  - 根拠: スペース padding は既存 `visual_column_to_char_index`（境界丸め）をそのまま使え変更が小さい。一方 tab 厳密一致は tab→スペース置換＝既存文字の replace を要し、前区間の `CompoundDiff`（delete 用の降順 insert 前提）に replace 系 sub-diff と順序・`reconcile_compound`・テストの波及が出るため変更ステップが多い、と実コードから見積もり。

### 残された曖昧さ（振る舞いレベル）

- なし（振る舞いゲートは閉じた）。grapheme 境界への丸め方向（CJK/tab 途中列で左右どちらの境界に寄せるか）と tab 分割展開は、C-4「基本の幅整合まで」の範囲として**明示的に実装裁量**へ委ねる（既存 `visual_column_to_char_index` の挙動に従う）。

### design へ送る実装論点（振る舞いではない）

- **clipboard の矩形メタ情報の持ち方**: 「直近の kill が矩形 kill か」を判定し、矩形 kill の各行を復元する機構。trait 拡張 / 内部の矩形 kill ring / 文字列へのマーカー埋め込み等。SystemClipboard 経由でも矩形性を保てるか（外部コピーとの往復）。→ openspec design の最初の論点。
- **複数行 insert の 1リビジョン化**: 前区間の `CompoundDiff` に挿入方向（`old=0, new>0`）の sub-diff を複数束ねる。delete は降順 remove だったが、insert も後ろの行から（降順で）適用し char index のずれを防ぐ必要。`CompoundDiff::reverse()` による undo が insert 方向で正しいか、`reconcile_compound` が insert 方向で正しいかの検証（insert 方向テストは現状なし）。
- **padding を含む行の diff 表現**: 短い行への「スペース padding＋本文」を、純粋 insert として1 sub-diff にまとめるか分けるか。
- **3変種のカーソル位置の算出**: 複数行挿入後にバッファが変形した後の最終カーソル位置の計算（左上は元位置、右下は最終行末尾、M-y は下端+1行）。

### 次にどうする

- requirement-definition へ渡して実装スコープに切り出す（振る舞いゲートは閉じた）。「clipboard の矩形メタ情報の持ち方」と「複数行 insert の 1リビジョン化（insert 方向の compound）」を、最初に潰すスコープとして切り出すことを推奨。
- C-5 の「パディング」側分岐は現状 paste 専用。次区間の string-rectangle も padding を要するなら、`_constraints.md` の C-5 を「パディング側も常駐」へ昇格させ、対象ごとの重複記述を畳む。
