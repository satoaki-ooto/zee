## 振る舞い定義: 矩形選択のハイライト

### 何のための対象か

ROADMAP「矩形のカット/削除まで」区間の表示側。
矩形選択中に、選ばれている (行 × 列) 領域が画面上で見て分かる状態にする。rectangle-selection の状態を、rectangle-mode.md の `text_style_at_char` / `draw_line` フローに反映する。

### 期待する振る舞い

濃度は散文＋要所 GWT。見た目の話だが、「どの visual column がハイライトされるか」は copy/cut の切り出し範囲と一致しなければならないので、その一点だけ厳密に固定する。

- 矩形選択が active な間、選択行範囲の各行について、矩形の visual column 範囲 `[left, right)` に重なる grapheme が selection 背景色でハイライトされる。
- **見た目と切り出しの一致（GWT）**
  - Given: 矩形が visual column `[left, right)`、行 r を含む
  - When: 行 r の各 grapheme を描画する
  - Then: その grapheme の visual column 区間が `[left, right)` と重なるものだけがハイライトされる。ハイライトされた範囲は、同じ行を copy/cut したときに切り出される範囲と**完全に一致**する（C-4）。
- **短い行（C-5: Emacs 準拠＝空扱い）**: 行末が `left` より手前の行は、その行ではハイライトされる文字が無い（行末より右の仮想空間は塗らない）。行末が `[left, right)` の途中までしか無い行は、行頭側の存在する文字だけがハイライトされる。
- 矩形モードを抜ける / キャンセルすると、ハイライトは消え通常表示に戻る。
- 通常の連続選択のハイライトは従来どおり（矩形と排他）。

### 許さない振る舞い・守る制約

- **絶対にこうならない**:
  - ハイライトされている列範囲と、実際に copy/cut される列範囲がずれる（C-4 違反。見た目が嘘になる）。
  - 矩形でない通常選択のハイライト（既存 `cursor.selection().contains()` 経路）の見え方が変わる（C-3）。
  - 行末より右の仮想空間を塗ってしまい、実在しない文字を選んでいるかのように見せる（C-5 の空扱いに反する）。
- **常に満たす前提**:
  - ハイライトは表示のみ。バッファ・カーソル状態を変更しない。
  - tab / CJK 幅をまたぐ grapheme の部分的ハイライト（grapheme の途中で列境界が来る）変則ケースは、この区間では基本整合まで（C-4 範囲、網羅は次区間）。

### 濃度

散文主体、「見た目＝切り出し範囲」の一致のみ GWT。理由: ハイライトは趣味的に詰めすぎない領域だが、見た目と実切り出しの不一致だけは致命的なので、そこだけ硬く固定する。

### 確定事項（ユーザ確認済み）

- **幅0矩形のハイライト**: 幅0でも選択状態は成立する（rectangle-selection で確定）ため、列位置に**細い縦線（カーソル様マーカー）**で示す。矩形挿入（次区間）の挿入位置の見せ方とも整合。

### 残された曖昧さ（振る舞いレベル）

- なし。

### design へ送る実装論点（振る舞いではない）

- `text_style_at_char()` は現状 `char_index` だけ受け取り `visual_x` を受け取っていない（rectangle-mode.md）。矩形列判定には描画側へ visual column を渡す必要があるが、これは実装手段。振る舞い要請は「列で判定してハイライトする」まで。
- アンカー端が画面外に出たときのスクロール挙動（`ensure_cursor_in_view` は range.start のみ参照、rectangle-mode.md）。最低限「対角カーソルが見える」ことは要請、アンカーも追うかは実装側裁量。

### 次にどうする

- selection / copy / cut と合わせて requirement-definition へ。
